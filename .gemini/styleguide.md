# Code Review Style Guide

## Language
- 모든 리뷰 코멘트, 요약, 제안은 반드시 한국어로 작성한다.
- 코드 식별자, 에러 메시지, 라이브러리 이름은 원문(영문)을 유지한다.
- 설명은 간결하고 실행 가능한 형태로 작성한다.

## Review Output Structure
- 리뷰 결과는 항상 아래 순서로 작성한다.
  1. `Summary`
  2. `Must Fix`
  3. `Questions`
  4. `Recommendations`
  5. `Good Points`
- 해당 섹션에 작성할 내용이 없으면 `없음`이라고 명시한다.
- `Must Fix`와 `Recommendations`를 섞지 않는다. 머지 전에 반드시 고쳐야 하는 문제만 `Must Fix`에 작성한다.

## Summary
- 변경의 의도와 영향 범위를 2-4문장으로 요약한다.
- 어떤 기능, 구조, 상태 관리, 라우팅, API 연동이 바뀌었는지 언급한다.
- 리뷰에서 집중한 위험 영역을 명시한다.

## Must Fix
- 실제 런타임 오류, 타입 오류, 빌드 실패, 데이터 유실, 인증 문제, 잘못된 라우팅처럼 머지 전에 반드시 수정해야 하는 항목만 작성한다.
- 각 항목은 아래 형식을 따른다.

```text
[severity] 파일 경로:라인
문제:
왜 문제인지:
수정 방향:
예상 영향:
```

### Severity
- `critical`: 앱 실행 불가, 빌드 실패, 인증 실패, 데이터 유실, 즉시 크래시
- `high`: 주요 사용자 플로우 실패, 잘못된 상태 저장, 라우팅 오류
- `medium`: 특정 조건에서 버그 가능성이 높거나 유지보수 리스크가 큰 문제
- `low`: 동작에는 문제가 없지만 수정 가치가 있는 문제

## Questions
- 코드만 보고 확정할 수 없는 설계 의도는 버그로 단정하지 않고 질문으로 분리한다.
- 서버 API 계약, 피그마 수치, 플랫폼별 UX, 오프라인 우선 정책처럼 맥락이 필요한 항목은 이 섹션에 작성한다.

## Recommendations
- 동작에는 문제가 없지만 구조, 타입 안정성, 재사용성, 접근성, 다국어, 플랫폼 분기 측면에서 개선하면 좋은 항목을 작성한다.
- 권장 개선 사항은 “반드시 수정해야 한다”처럼 표현하지 않는다.
- 단순 취향, 포맷팅, 네이밍 선호는 린트 또는 프로젝트 컨벤션 위반이 아니라면 이 섹션에만 작성한다.

## Good Points
- 좋은 구조적 선택, 타입 안정성, 재사용성, 접근성 대응, 컨벤션 준수 등을 구체적으로 언급한다.
- 단순 칭찬보다 어떤 코드 선택이 왜 좋은지 짧게 설명한다.

## Review Rules
- 현재 diff와 관련 파일의 실제 최신 내용을 기준으로 리뷰한다.
- 이전 버전 코드나 추측에 기반한 리뷰를 하지 않는다.
- `return` 누락, import 경로 오류, 상태 필드 누락 같은 지적은 반드시 현재 파일 전체를 확인한 뒤 작성한다.
- 확실하지 않은 항목은 `Must Fix`가 아니라 `Questions` 또는 `Recommendations`로 분류한다.
- 리뷰 코멘트에는 가능한 한 파일 경로와 구체적인 코드 위치를 포함한다.
- 제안 코드는 현재 프로젝트 구조와 컨벤션을 따라야 한다.
- `src/features` 경로는 새 코드 제안에 사용하지 않는다. 화면은 `src/screens`, 상태는 `src/state`, 재사용 컴포넌트는 `src/components`, 공용 훅은 `src/hooks` 기준으로 제안한다.
- UI 컴포넌트는 가능한 `src/components/ui`의 base primitive를 기반으로 확장하도록 제안한다.
- i18n 리뷰에서는 모듈 레벨 `t()` 호출, 하드코딩 문자열, locale 변경 반영 가능성을 확인한다.
- Expo/React Native 리뷰에서는 iOS와 Android 동작 차이를 함께 고려한다.
